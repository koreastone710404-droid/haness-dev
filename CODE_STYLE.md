# Code Style Guide

## 목적

이 문서는 `Haness Engineering` 팀의 코드 품질과 일관성을 유지하기 위한 기준을 제공합니다.

## 기본 원칙

- 가독성, 유지보수성, 일관성을 최우선으로 합니다.
- 의미 있는 이름과 명확한 책임 분리를 적용합니다.
- 복잡한 로직은 작은 단위로 나누고, 읽기 쉬운 구조로 작성합니다.

## 포맷 규칙

- 들여쓰기: 프로젝트에 따라 스페이스 2개 또는 4개를 일관되게 사용합니다.
- 줄 길이: 100자 이내를 권장합니다.
- 빈 줄: 논리적 단위마다 적절히 구분합니다.
- 코드 블록, 제어문, 함수 선언은 일관된 스타일로 작성합니다.

## 네이밍 규칙

- 변수/함수
  - JavaScript/TypeScript: `camelCase`
  - Python: `snake_case`
- 클래스/타입: `PascalCase`
- 상수: `UPPER_SNAKE_CASE`
- boolean값 변수는 `is`, `has`, `should`, `can` 등의 접두어를 사용합니다.
- 도메인 용어를 반영하여 의미 있는 이름을 사용합니다.

## 구조 및 모듈화

- 하나의 파일은 하나의 책임에 집중합니다.
- 공통 로직은 재사용 가능한 함수 또는 유틸리티로 분리합니다.
- 컴포넌트/모듈 간 의존성은 최소화하고, 순환 의존성을 피합니다.

## import/의존성 관리

- import는 그룹화하여 정렬합니다.
  1. 외부 패키지
  2. 내부 모듈
  3. 스타일/자산
- 사용하지 않는 import는 제거합니다.
- 상대경로 대신 절대 경로나 별칭을 사용하는 규칙이 있다면 일관되게 적용합니다.

## 예외 처리

- 예외는 필요한 경우에만 사용하고, 사용자에게 친절한 메시지를 제공합니다.
- 입력값 검증과 경계값 처리를 코드 작성 초기부터 적용합니다.
- 오류가 발생할 수 있는 지점은 명확히 처리하고, 로그 정보를 충분히 남깁니다.

## 주석 및 문서화

- 주석은 구현이 아니라 의도를 설명하는 데 사용합니다.
- 복잡한 비즈니스 규칙이나 중요한 설계 결정에는 주석을 추가합니다.
- TODO 주석 작성 시 이유와 해결 시점을 함께 적습니다.

## 테스트 코드 스타일

- 테스트 이름은 수행 목적을 명확히 표현합니다.
- 단위 테스트는 독립적으로 실행될 수 있어야 합니다.
- 통합 테스트는 실제 시나리오를 반영하고 재현 가능한 데이터를 사용합니다.
- 테스트 파일명은 언어/프레임워크 규칙에 따라 일관되게 작성합니다.

## 린트 및 자동 포맷

- 프로젝트에 지정된 린트 및 포맷 도구를 사용합니다.
- 커밋 또는 PR 생성 전에 자동화 도구를 실행합니다.
- 예시 도구
  - JavaScript/TypeScript: ESLint, Prettier
  - Python: Flake8, Black, isort
- 설정 변경 시 팀과 사전 협의를 진행합니다.

## 커밋 메시지 권장 스타일

- 커밋 메시지는 변경 의도를 명확히 설명합니다.
- 형식 예시: `feat: 로그인 API 추가`, `fix: 응답 예외 처리 수정`
- 필요 시 다음 항목을 포함합니다.
  - 무엇을 변경했는지
  - 왜 변경했는지
  - 테스트/검증 방법

## 지속적 개선

- 코드 스타일은 팀 합의에 따라 주기적으로 개선합니다.
- 새 규칙 추가 또는 변경 시 문서를 함께 업데이트합니다.
- 코드 리뷰에서 스타일 관련 피드백은 교육적 관점에서 제공합니다.
