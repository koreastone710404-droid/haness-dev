# Harness Engineering

이 skill은 **모델 프롬프트를 길게 쓰는 일**보다 **에이전트가 일하는 환경을 설계하는 일**에 집중한다. 목표는 다음 세 가지다.

1. 에이전트가 필요한 정보를 저장소 안에서 스스로 찾을 수 있게 한다.
2. 품질 규칙을 사람 리뷰어에만 의존하지 않고 기계적으로 강제한다.
3. 실패를 다음 실행의 개선 재료로 바꾸는 피드백 루프를 만든다.

## 언제 사용하는가

다음 상황이면 이 skill을 우선 사용한다.

- 새 프로젝트에 Claude Code, Codex, Cursor, Copilot 같은 코딩 에이전트를 본격 도입하려고 한다.
- 이미 에이전트를 쓰고 있지만 산출물이 들쭉날쭉하고, 같은 실수가 반복된다.
- 규칙이 Slack, Notion, 회의록, 사람 머릿속에 흩어져 있어 에이전트가 맥락을 자꾸 놓친다.
- `CLAUDE.md`, `AGENTS.md`, `.claude/skills/`, `docs/`, 테스트/린트/eval 체계를 어떻게 나눌지 정해야 한다.
- "프롬프트를 더 잘 쓰면 해결될 것 같다"는 감각이 들지만 실제로는 구조 문제일 가능성이 높다.

다음 상황이면 이 skill을 **보조적으로만** 사용한다.

- 단일 함수 수정 같은 하네스 설계와 무관한 작은 구현 작업
- 현재 구조 감사 없이 바로 코드만 빨리 고치면 되는 긴급 핫픽스

## 핵심 원칙

### 1. 사람은 방향을 정하고, 에이전트는 실행한다

사람의 핵심 역할은 직접 모든 코드를 쓰는 것이 아니라, 목표·제약·검증 기준·피드백 경로를 설계하는 것이다. 구현 세부는 가능한 한 에이전트가 담당하게 둔다.

### 2. 저장소를 단일 진실 원천으로 만든다

에이전트 입장에서 **실행 중 접근할 수 있는 지식만 아는 지식**이다. 핵심 규칙, 아키텍처 결정, 제품 요구사항, 운영 절차는 버전 관리되는 저장소 안으로 옮긴다. `CLAUDE.md` 또는 `AGENTS.md`는 백과사전이 아니라 **짧은 안내 지도**로 유지하고, 상세 내용은 `docs/`와 개별 skill/reference 파일로 분리한다.

### 3. 읽기 쉬운 코드보다 먼저, 에이전트가 읽기 쉬운 시스템을 만든다

사람이 한 번 이해하기 쉬운 코드만으로는 부족하다. 에이전트가 반복 실행에서도 빠르게 위치를 찾고 속도를 복원할 수 있게 다음을 명시한다.

- 디렉터리 책임
- 레이어 경계
- 의존 방향
- 용어집
- 주요 실행 경로
- 검증 기준

### 4. 구현이 아니라 불변식을 강제한다

어떤 라이브러리를 반드시 쓰라고 강요하기보다, 어떤 제약을 항상 지켜야 하는지 강제한다. 예:

- 경계에서 입력을 검증한다
- 로깅을 구조화한다
- 계층 역참조를 금지한다
- 네트워크/DB 접근은 정해진 레이어에서만 한다
- 공개 API shape은 테스트에 화석으로 고정한다

구현 선택의 자유는 남기되, 시스템 일관성은 린트·테스트·정적 검사로 지킨다.

### 5. 문서만 쓰지 말고 실행 가능한 날을 만든다

같은 명령, 같은 검증 스크립트, 같은 리포트 생성 코드를 반복해서 쓰게 하지 않는다. 반복되고 결정적인 작업은 `scripts/`로 옮겨 실행하게 한다.

### 6. 실패를 기록 가능한 피드백으로 바꾼다

버그, 리뷰 코멘트, 뒤틀린 PR, 반복되는 에이전트 실수는 흩어 두지 말고 다음 중 하나로 흡수한다.

- 문서 규칙 추가
- skill 수정
- lint/test 추가
- eval 케이스 추가
- 템플릿/스크립트 보강

## 시작 시점에 반드시 할 일

1. 현재 저장소 구조를 빠르게 감사한다.
2. 에이전트가 자주 실패하는 지점을 최소 3개 찾는다.
3. 실패 원인을 아래 네 범주로 분류한다.
   - 맥락 부족
   - 구조 모호함
   - 검증 부재
   - 피드백 미반영
4. "프롬프트 개선"으로 충분한 문제와 "하네스 수정"이 필요한 문제를 분리한다.
5. 변경 범위를 **작게 시작**한다. 처음부터 모든 것을 재설계하지 않는다.

## 기본 워크플로우

### Phase 0. 시작 프레이밍

다음을 짧게 정리한다.

- 목표 작업: 에이전트가 수행해야 하는 대표 작업 3~5개
- 현재 실패 패턴: 예) 레이어 위반, 문서 누락, 테스트 미작성, 잘못된 경로 참조
- 성공 기준: 예) 테스트 통과율, 리뷰 수정 횟수, 되돌림 감소, 완료 시간 단축

### Phase 1. 현황 감사

다음 파일과 디렉터리를 우선 점검한다.

- `CLAUDE.md`
- `AGENTS.md` 또는 에이전트 정의 파일
- `.claude/skills/`
- `docs/`
- `scripts/`
- `tests/`
- CI 설정
- 린터/포매터/타입체커 설정

감사 결과를 아래 형식으로 정리한다.

```md
## Audit Summary
- System of record: 명확 / 부분적 / 없음
- Architecture map: 있음 / 부족 / 없음
- Skill coverage: 충분 / 일부 / 없음
- Mechanical checks: 강함 / 보통 / 약함
- Feedback loop: 있음 / 약함 / 없음
- Highest-risk gaps:
  1. ...
  2. ...
  3. ...
```

### Phase 2. 정보 구조 설계

다음 원칙으로 정보를 재배치한다.

- `CLAUDE.md`: 짧은 지도. 작업 방식, 핵심 규칙, 상세 문서 포인터만 둔다.
- `AGENTS.md` 또는 에이전트 파일: 역할, 책임, 핸드오프 규칙을 둔다.
- `docs/`: 사람이 읽고 에이전트도 참조할 수 있는 상세 문서를 둔다.
- `.claude/skills/`: 반복 작업과 절차 지식을 둔다.
- `scripts/`: 결정적이고 반복적인 실행 코드를 둔다.

권장 예시:

```text
.
├── CLAUDE.md
├── AGENTS.md
├── docs/
│   ├── architecture/
│   ├── product/
│   ├── runbooks/
│   ├── decisions/
│   └── evals/
├── scripts/
├── tests/
└── .claude/
    └── skills/
        ├── harness-engineering/
        ├── verification-loop/
        ├── code-review/
        └── release-check/
```

### Phase 3. 불변식 인코딩

문서 규칙을 가능한 한 코드로 옮긴다.

우선순위가 높은 대상:

1. 레이어 의존 방향 검사
2. 입력/출력 스키마 검증
3. 구조화 로깅 규칙
4. 테스트 누락 탐지
5. 파일/모듈 크기 제한
6. 금지 import 또는 금지 API 사용

문서에만 적고 끝내지 않는다. 가능하면 다음으로 바꾼다.

- ESLint / Ruff / golangci-lint / ktlint 규칙
- 구조 테스트
- 계약 테스트
- CI 체크
- pre-commit / pre-push 훅

### Phase 4. Skill 설계

새 skill을 만들거나 기존 skill을 고칠 때는 다음을 따른다.

1. `name`은 짧고 검색 가능한 타이포 한글 이름으로 정한다.
2. `description`에는 **무엇을 하는지**와 **언제 써야 하는지**를 함께 넣는다.
3. 본문은 500줄 수준을 넘기지 않도록 유지한다.
4. 길어지면 `references/`로 분리하고, 본문에는 "언제 어떤 파일을 읽을지"만 적는다.
5. 반복 코드가 생기면 `scripts/`로 뺀다.

설명을 모호하게 쓰지 않는다.

나쁜 예:

```yaml
name: project-helper
description: 프로젝트를 도와준다.
```

좋은 예:

```yaml
name: verification-loop
description: 구현 후 검증 루프를 수행한다. 테스트 실행, 실패 분류, 로그/출력 확인, 회귀 샘플 점검, 수정 후 재검증이 필요할 때 사용한다. "verify", "검증", "테스트 돌려", "회귀 확인" 같은 요청이 나오면 이 skill을 사용한다.
```

### Phase 5. Eval과 피드백 루프 구축

대화 원격에 작게 고정해 eval 세트를 만든다.

예:

- 새 API 엔드포인트 추가
- 기존 버그 재현 후 수정
- 문서 변경에 반응한 리팩터링
- 실패 로그를 보고 원인 추적
- 테스트가 없는 모듈에 최소 수준만 추가

각 eval에는 다음을 남긴다.

- 입력 요약 설명
- 기대 결과
- 실패 시 힌트
- 적용된 하네스 수정
- 재실행 결과

### Phase 6. 결과 보고

최종 결과는 반드시 **구조 변경 + 검증 방법 + 남은 리스크**를 함께 보고한다.

다음 형식을 기본으로 사용한다.

```md
## Harness Update
### Changed
- ...

### Why
- ...

### New Invariants
- ...

### New/Updated Skills
- ...

### Eval Results
- task A: pass/fail
- task B: pass/fail

### Remaining Risks
- ...

### Next Smallest Improvement
- ...
```

## 산출물 체크리스트

가능하면 아래 산출물을 만든다.

- `CLAUDE.md` 또는 동등한 루트 가이드
- 아키텍처 맵 문서
- 핵심 용어집 또는 도메인 맵
- 1개 이상의 검증 skill
- 1개 이상의 반복 작업용 script
- 최소 eval 세트
- 변경 이력

## 판단 규칙

### 프롬프트를 늘릴지, 하네스를 고칠지

다음이면 하네스를 고친다.

- 같은 실수가 반복된다.
- 새 세션마다 다시 설명해야 한다.
- 규칙은 인간 리뷰가 매번 수동으로 잡는다.
- 정답이 있는데 저장소 안에 없다.
- 구현보다 경계/구조 위반이 더 자주 발생한다.

다음이면 프롬프트 보강만으로도 충분할 수 있다.

- 일회성 혹은 요청이다.
- 저장소 구조는 이미 잘 정리되어 있다.
- 검증 체계도 충분하고, 단지 요청 표현이 약간 모호하다.

## 주의사항

- 장문의 단일 매뉴얼로 모든 것을 해결하려 하지 않는다.
- 사람만 이해하는 문서 구조를 만들지 않는다.
- 규칙을 README 한 곳에만 숨기지 않는다.
- "이건 상식이니까 적지 않아도 된다"라고 가정하지 않는다.
- time-sensitive 정보는 외부 링크에만 의존하지 않는다. 핵심 판단 근거는 저장소에 복사·요약한다.
- 하네스 변경은 기능 개발과 분리해 작은 단위로 둔다. 한 번에 너무 많이 바꾸면 효과를 측정할 수 없다.

## 종료 조건

다음이 충족되면 이번 작업은 충분하다.

1. 대표 실패 패턴 중 최소 1개 이상이 문서가 아니라 **시스템 변경**으로 해결되었다.
2. 새 규칙이 사람이 아니라 도구/테스트/구조에 의해 강제된다.
3. 다음 세션의 에이전트가 추가 설명 없이 같은 작업을 더 잘 수행할 수 있다.
4. 남은 리스크와 다음 개선 단위가 명확히 기록되었다.

이 skill의 기본 태도는 "더 좋은 프롬프트"보다 "더 좋은 환경"을 만드는 것이다. 항상 가장 작은 구조적 개선부터 시작한다.
